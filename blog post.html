Test-driven development - what is it?

How does that possibly apply to archives?

What about the overhead?

The Bentley's EADs cover over 80 years of changing descriptive practice, and represent literally thousands of hours of work on the part of the staff and students who were involved in their creation. The last thing we want to do while automating our xml normalizations is make an unintended change that nullifies the work of those who came before us. Unfortunately, uncertainty is always a factor when working with messy data. [more about small changes having unintended effects]

The solution outlined here is a practice that is nearly ubiquitously adopted in the world of software development, especially those that follow agile development methods, and when applied properly [...]

Even if you're not familiar with programming languages, python is unusually readable, so follow along and I think you'll be surprised at how much it makes sense!

The mantra of TDD: test first, then code. So, let's make a new file to hold all our test code (we'll call it test_date_normalizer.py) and start writing:

# first, we'll define the original input text, something we
# might find in an average EAD <unitdate> tag,
# then the text we want to transform the original into
date_text_raw = "July 1, 2014"
date_text_standard = "2014-07-01" 

# run the (currently unwritten) code to transform the input text,
# and store the result in a new variable
normalized_date_text = normalize_date(date_text_raw) 

# test to see whether the result is exactly equal to our desired output
assert date_text_standard == normalized_date_text

now run it and...

Ok, that's a little pedantic (the code doesn't actually do anything with the input string, it just cheekily returns the string we want), but it really is important to do these small, path-of-least-resistance edits, especially as we are just learning the concept of TDD. Small, iterative steps are incredibly important when building anything complex - keeps things [].
